# AI Usage Toolbar — Engineering Design

This document is the v1 engineering plan. It tracks the modules, data flow, and concrete trade-offs that the implementation should follow. The product context is in `PRODUCT.md`.

## Stack

- **Language / UI:** Swift 6, SwiftUI for the popover, AppKit's `NSStatusItem` (via `MenuBarExtra` + `MenuBarExtraAccess`) for the menu bar host.
- **Minimum deployment target:** macOS 14 (Sonoma). `MenuBarExtra` is macOS 13+, but Swift Charts on 14 is meaningfully better and most existing prior art targets 14+.
- **Project layout:** Generated by XcodeGen from `project.yml`. We commit the YAML and gitignore the `.xcodeproj` so the source of truth stays diffable.
- **Auto-update:** Sparkle (added in v1.1, not v1).
- **Charts:** Swift Charts (Apple) for daily/weekly sparklines.
- **Dependencies:** kept to two — `MenuBarExtraAccess` (status-item escape hatches) and `Sparkle` (later). Everything else is stdlib.

## Data flow

```
[~/.claude/projects/**/*.jsonl]
            │
   FileMonitor (FSEvents, debounced 250ms)
            │
   ClaudeJSONLReader  ── streams lines, decodes, dedupes by (messageId, requestId)
            │
   UsageAggregator    ── current 5h block, weekly sum, today sum, 7-day buckets
            │
   AppState (@Published UsageSnapshot)
            │
            ├── MenuBarLabel   (just the % string + color)
            └── PopoverRoot    (full card stack)
```

A `Task` owned by `AppState` consumes events from the monitor's `AsyncStream`. On every change a fresh `UsageSnapshot` value is published; SwiftUI diffs it.

## JSONL parser specifics

- Glob both `~/.claude/projects/` and `~/.config/claude/projects/`, honoring `CLAUDE_CONFIG_DIR` (comma-separated list of overrides).
- Each line is a JSON object. We only care about lines where `message.usage` is present (assistant turns).
- Token total per turn:
  `input_tokens + cache_creation_input_tokens + cache_read_input_tokens + output_tokens` — `input_tokens` alone is the post-cache-breakpoint count and undercounts by 5–15% on cached prompts.
- Dedupe key: `"\(message.id):\(requestId)"`. Without this we double-count when Claude Code occasionally writes duplicate assistant lines (confirmed in ccusage's source).
- Carry `cwd`, `gitBranch`, `model`, `timestamp` on every event so we can group by project in v1 and per-model in v1.1.

## 5-hour block algorithm

Lifted from ccusage. Sort all events chronologically. A new "block" starts on the first event whose timestamp is more than 5 hours after the previous event's timestamp, or at the start of input. The active block is the one whose start is within the last 5 hours; remaining time = `(blockStart + 5h) - now`.

## Plan limits

Thresholds are not exposed by any API. We hardcode a `PlanLimits` enum:

| Plan  | 5h tokens | Weekly tokens | Weekly Opus tokens |
| ----- | --------- | ------------- | ------------------ |
| Pro   | ~44k      | undocumented  | n/a                |
| Max5  | ~88k      | computed      | computed           |
| Max20 | ~220k     | computed      | computed           |

These numbers drift; the docs are silent. Strategy: start with Claude-Code-Usage-Monitor's published constants, expose an override in Settings, and detect plan from `~/.claude/settings.json` if a field surfaces (we'll watch for one).

## File watching

`FSEventStream` on the projects root, latency 250 ms, flags `useCFTypes | fileEvents`. Wrapped in an `AsyncStream<URL>` so it composes with Swift concurrency. Fallback polling every 60 s in case FSEvents drops events under sandbox or external-disk conditions.

## Notifications

`UNUserNotificationCenter` requests at first launch (deferred until first 75% threshold to avoid scaring new users). Three categories: `warn-75`, `warn-90`, `warn-95`. Each carries a "Snooze until reset" action that suppresses further alerts for the active block.

## Pricing table

Vendor a `pricing.json` file mirroring LiteLLM's model price keys. Sample shape:

```json
{
  "claude-opus-4-7": { "input_per_mtok": 15, "output_per_mtok": 75, "cache_write_per_mtok": 18.75, "cache_read_per_mtok": 1.5 },
  "claude-sonnet-4-6": { "input_per_mtok": 3, "output_per_mtok": 15, "cache_write_per_mtok": 3.75, "cache_read_per_mtok": 0.3 }
}
```

These numbers drift; we ship a refresher script (v1.1) that pulls the current values from LiteLLM's repo. For v1, vendored values are fine.

## What v1 ships

- Menu-bar label with template icon + percent string, color-shifting at thresholds.
- Popover with three cards as designed in PRODUCT.md.
- File-watching parser with dedupe + 5h blocks.
- Pricing table for cost estimates.
- Notifications at 75 / 90 / 95.
- Settings: which metric shows inline (% / time / $), plan tier override.

## What v1 explicitly defers

- Admin API integration.
- OpenRouter / OpenAI integration.
- Per-project breakdown view (data is captured, view is deferred).
- Predictive rate-limit warning (the differentiator from PRODUCT.md — high-priority v1.1).
- Sparkle auto-update wiring.
- Mac App Store distribution.

## Performance budget

- Cold start to first snapshot: under 1 second on a corpus of 100 MB of JSONL.
- Idle RAM: under 60 MB.
- CPU at idle (popover closed, file activity quiet): effectively 0% — pure FSEvents wait.
- CPU during burst (active Claude Code session writing every few seconds): under 2% on M-series.

## Testing

- Unit tests on `ClaudeJSONLReader` and `UsageAggregator` with fixture JSONL files (5h block boundary, duplicate messageId, cache token math, model switching mid-session).
- A manual checklist for menu-bar rendering across light/dark mode, three macOS scaling levels, and the macOS 26 Liquid Glass tray treatment.
- Snapshot tests on the popover at the three theme/size combinations using `swift-snapshot-testing`.

## Risks

- **Plan thresholds drift.** Mitigation: settings override + watch upstream tools' constants.
- **Pricing drift.** Mitigation: vendored JSON, refresh script.
- **Sandbox friction reading `~/.claude/`.** Mitigation: ship Developer-ID-signed, not sandboxed, in v1. Revisit if we pursue Mac App Store.
- **Crowded market.** Mitigation: lean hard on the v1.1 predictive-warning differentiator from PRODUCT.md.
